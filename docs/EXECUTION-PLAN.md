# Vibemesh-AI — Execution Plan

*Turns the [R&D study](RND-GENERALIZATION-AND-SKILLS-STUDY.md) into a dependency-sequenced, bench-gated, codebase-grounded build plan. Produced by a senior-lead-led R&D board (skeleton → per-phase detail → sequencing). Phase `p0` detail authored by the orchestrator after that one agent timed out; all else is the board's output.*

## Overall approach

Ship the R&D study's "general skeleton, generalize the frozen brain" thesis as a dependency-sequenced, bench-gated migration of exactly one seam (contextText → assembleSystemPrompt) plus a thin app-side understanding/verification layer — never a rewrite. Sequencing is forced by three hard facts grounded in the code: (a) the bench:gate is only trustworthy once BENCH_SAMPLES median aggregation is the standard run mode (its IoU tolerances are deliberately wide because a single run hits a non-deterministic live API), so multi-sample is a precondition for every "zero regression" claim; (b) the spine/skills refactor hard-blocks all mechanism work, so it lands before any skill is authored; (c) composition (multi-skill → ONE Customizer block + a correctly-mated all-view) is the big unaddressed gap and depends on ≥2 skills existing, so it lands after the registry buildout. I front-load two cheap, independent risk reducers that protect every later phase regardless of the skills work: an app generation timeout (none exists today — runGeneration wraps streamGenerate in an AbortController with no time bound) and the mandatory-code-block contract fix (both cloud engines hit the contractViolated drop at store.ts:477). The honest ceiling is held throughout: organic/figurative stays ~60-70% on rotate_extrude/polygon, the 'mechanism understanding' is a curated finite menu wearing a registry costume, and tool-use is unavailable across engines so skills are system-prompt injection only — budget-aware assembly is mandatory, capped by the smallest engine's window (local num_ctx). Phase 0 banks the already-done genericization work (the uncommitted prompt diff generalizes Hard-surface into faceted/low-poly with an ARCHETYPE+FACET-VERDICT split — it does NOT re-add the chess 6-piece table the study feared; chess survives only as one inline parenthetical) so later phases build on a clean, committed tree.

## Guardrails (non-negotiable across every phase)

1) RESPONSE CONTRACT IS SACRED: every change preserves prose + exactly ONE ```scad fenced block starting with a Customizer parameter block (extractScadBlock/parseParameters depend on it) and the printability invariants (manifold, flat on z=0, mm, ≥1.2mm walls, no global $fn, no import/text()/libraries). The openscad-contract skill governs any edit to prompt.mjs / params.ts / the render pipeline / export paths. 2) BENCH-GATE EVERY GEOMETRY/PROMPT CHANGE: npm run bench → npm run bench:gate, and the multi-sample median run mode (Phase 0) must be standard before trusting any 'zero regression' claim — compiled/overSplit stay zero-tolerance. 3) NO USE-CASE HARDCODING: new knowledge goes into registry fragments selected at the seam, NEVER into a SYSTEM_PROMPT edit; the abstract spine stays engine-agnostic and must fit local num_ctx. 4) PORTABILITY: preserve the cache_control ephemeral split on anthropic, the Kimi plain-string variant (no thinking/cache_control), the claude-code single-turn flatten + 4-image CAP, and the local exemplar-drop budget rule. 5) VALIDATORS WARN/REFINE, NEVER HARD-BLOCK in the live loop (mirroring structuralReport); only the deterministic zero-API CI walker may hard-fail. 6) HONEST CEILING: do not market open-ended mechanism understanding or chase the organic lane on OpenSCAD; name kernel limits (B-rep/STEP, true threads, kinematics) rather than implying they're roadmap items. 7) SSE WIRING INTACT: keep res.on('close') + AbortController upstream-abort on the generate route. 8) ADVISORY-ONLY UNDERSTANDING: the intent object and applied-patterns chip are context/metadata, never a competing output block, and must version WITH the code on each ChatMessage so rollback never shows stale metadata. 9) Update docs/SPEC.md whenever image/refine/versioning/multi-part behavior changes.

### p1-robustness-floor — Robustness foundation: gen-timeout, contract-fix, first-render floor

**Goal.** Close three reliability gaps that are independent of the skills work and protect every later generation: an app-side generation timeout, a hardened mandatory-code-block recovery, and a quality floor on the first render.

**Why now.** runGeneration (store.ts:452) wraps streamGenerate in an AbortController with NO time bound — only the render watchdogs (RENDER_TIMEOUT_DRAFT/EXPORT) exist, so a stalled stream hangs the UI indefinitely. The contractViolated drop (store.ts:477, code===null||blockCount>1) is hit by both cloud engines and only nudges within MAX_AUTO_FIX=2; the study flags this as a robustness gap to harden (clearer re-ask, telemetry, interaction with the abort path). A first-render quality floor catches the degenerate-but-clean case that degenerateReason already half-detects. These are cheap, high-leverage, and unblock nothing downstream — landing them early de-risks the noisier phases.

### Cross-phase dependencies

P0 is the universal precondition: the multi-sample median gate it establishes is the ONLY trustworthy 'zero regression' bar for every later phase (the gate's IoU/quality tolerances are deliberately wide because a single run hits a non-deterministic live API), and it banks the uncommitted tree so no later phase conflates its effect with prompt drift. P1 depends ONLY on P0 (the gate must exist to validate the three reliability changes) and is independent of P2-P7 — it can run in parallel with the start of P2 since it touches app-side reliability only (no prompt.mjs / contextText seam edits). P2 is the architectural choke point: it generalizes the single seam (contextText -> assembleSystemPrompt), splits prompt.mjs into spine+fragments, introduces selectSkills + the keyed registry + the SSE done.skillIds field + the zero-API registry walker, and HARD-BLOCKS P3, P4, and P7 (no skill can be authored before the registry exists; re-hardcoding the seam pre-P2 is a guardrail violation). P3 depends on P2 and is the de-risking spike that produces the shared fit() helper + the copyable authoring template + the first walker entry that P4 consumes. P4 depends on P3 (template + fit() helper + walker pattern) and P2 (assembler + walker + interferenceScore-on-generated-parts); within P4 the ordering is forced by render budget and correctness risk (native-easy -> native-hard gears -> budget-gated coil-spring/routed). P5 is hard-blocked by P2 (assembleSystemPrompt + selectSkills + GenerateContext.skillIds + SSE done return path) and benefits from P4 (real skills to attribute on the chip; degrades gracefully to intent-only with empty skillIds). P6 has one internal precondition (estImageTokens pixel-scaling must land FIRST or tiling will mislead the history budget) and depends on P5 for the inline-intent path it extends; its refine-gate rework (programmatic proxy primary, VLM advisory) is independent of skills. P7 lands LAST: it depends on BOTH P2 (assembler + walker + paramAliases schema) AND P4 (>=2 selectable mechanism skills must exist before composition can be exercised). Cross-cutting through all of P2-P7: every prompt/geometry/seam edit is openscad-contract-governed and bench-gated under the P0 multi-sample mode; the zero-API walker is the only thing allowed to hard-fail, all live-loop validators WARN/refine only; SPEC.md is updated whenever image/refine/versioning/multi-part behavior changes (P5, P6, P7).

### Top risks

- **LIVE-API NONDETERMINISM makes 'zero regression' unfalsifiable. The bench hits a non-deterministic live API and bench:gate's IoU/quality tolerances are deliberately wide for exactly this reason — a single run can pass while a real quality regression slips through, and every phase from P2 on claims 'no regression vs baseline'.** → Make BENCH_SAMPLES median aggregation the STANDARD run mode in P0 (compiledRate + median quality scores) before any prompt/seam/skill change, and only then tighten tolerances on the sampled metrics. compiled/overSplit stay zero-tolerance. No phase may claim 'zero regression' off a single-sample run. Re-seed baseline.json via gate --update-baseline only when a metric legitimately shifts, with committed rationale.
- **LARGE-OUTPUT-AGENT / EXEMPLAR-POISON: skills are system-prompt few-shot injection (tool-use is unavailable across all engines), so a subtly-wrong exemplar is copied structurally and silently degrades EVERY generation that selects it — the live bench scores it a clean buildability=1.0 with no note. Composition compounds this: two full exemplars can blow the smallest engine's window (local num_ctx) and a naive concat truncates the Customizer header (params.ts ASSIGN_RE stops at the first non-assignment line), silently dropping sliders.** → The zero-API registry walker (compile + buildability + interference + per-skill validator + broken-control discrimination) is the NON-OPTIONAL hard gate for every skill before any baseline bump; each skill carries a version + auto-quarantine-on-compile-failure so selectSkills never injects a broken few-shot. The assembler keeps the greedy per-engine token cap and degrades to directive-only (never truncates mid-exemplar); paramAliases collapse shared concepts to one parameter and the walker asserts every alias names a real parameter in that skill's header.
- **RENDER BUDGET: openscad-wasm is single-shot (fresh instance per render) under a 90s watchdog with one auto-retry-at-Draft; the involute spur-gear point count and especially the coil-spring helix are heavy and can blow the watchdog at higher quality presets, and a printed micro-coil binds on FDM anyway. A naive wall-clock generation timeout would also kill legitimately-slow high-effort Anthropic/claude-code streaming.** → Order P4 by render cost (native-easy prismatic/boolean first — the Manifold sweet spot). Coil-spring is NOT authored on schedule — only per a measured end-to-end (render+STL+r3f+interactive-nudge) verdict across all four presets; button-return routes to a metal-spring SEAT by default. The P1 generation timeout is an INACTIVITY (idle-stream) watchdog re-armed on every delta (mirroring bench/run.mjs req.setTimeout(0)+long killer), per-engine-tunable, longest/off for claude-code — never a total cap.
- **OVER-SELECTION cascade: selectSkills is detectKitIntent x N skills, multiplying the over-trigger surface — a plain 'wheel' or 'bracket' prompt could light a mechanism skill (or many), poisoning a simple part with irrelevant few-shot and, at >=2, spuriously firing the composition directive.** → A mandatory negative-case bench lane (plain bracket/knob/solid-wheel -> ZERO mechanism skills) is in bench:gate from P2 onward and fails the build if it ever selects a mechanism skill; selectors require paired nouns (wheel AND axle/shaft, not 'wheel' alone); detectKitIntent's existing kit/spinner calibration is preserved, not rewritten; the composition directive only emits at >=2 selected skills so the negative lane staying green protects it too.
- **RESPONSE-CONTRACT FRAGILITY: extractScadBlock/parseParameters depend on prose + exactly ONE scad-tagged fenced block starting with a Customizer header. The P5 INTENT preamble, the P6 vision fragments, and the P7 merged header all risk tripping the blockCount>1 contract drop or truncating the slider list — and prompt.mjs / params.ts / render / export edits are the highest-blast-radius changes in the repo.** → All such edits are governed by the openscad-contract skill and bench-gated. The INTENT preamble is a non-fenced single line (blockCount keys only on scad/openscad fences — verified against the regex), stripped from displayed prose; the bench asserts blockCount===1 alongside intent emission; the merge directive reconciles duplicate Customizer rows to one parameter per concept; degenerateReason + the first-render floor backstop any header that still parses degenerately.
- **UNDERSTANDING-LAYER ENTRENCHMENT + STALE-METADATA UX bugs: a confidently-wrong intent object can lock in an error a plain single-pass would self-correct, and metadata stored on a global 'latest' field would show stale applied-patterns after a rollback (the existing per-message restorable versions + vpPast/vpFuture make this a real, study-flagged bug). The refine loop today also re-feeds the same render to the SAME model — self-grading, not verification.** → Intent stays strictly ADVISORY (it only seeds selectSkills + visible PLAN lines; the model still reads the actual request) and every wrong chip has one-click corrected re-generation; intent + appliedSkillIds live ON the ChatMessage and ride restoreVersion/restoreNewer/persist so rollback shows that version's metadata or none. In P6 the programmatic IoU/bbox proxy (truly model-independent, zero-cost) becomes the PRIMARY refine gate and the VLM critic is demoted to an advisory tie-breaker.
- **HONEST-CEILING DRIFT: the spike's easy win (wheel-axle) and the registry's framing could imply open-ended mechanism understanding, when in reality it is a curated finite menu wearing a registry costume — gears are an involute APPROXIMATION (rack-cutter, not true involute), springs are native-hard with material caveats, organic/figurative stays ~60-70% on rotate_extrude/polygon, and the kernel cannot do B-rep/STEP, true threads, or kinematics.** → Name kernel limits explicitly in docs and skill fragment text (material caveats on living-hinge/spring/gear, insert-advice on bearing-pocket) rather than implying they are roadmap items; a vision critic makes the organic ceiling measurable, not higher. Do not market arbitrary assembly — composition is bounded to mechanical kits sharing datum/fit primitives. The spike proves the PLUMBING, not that hard mechanisms are cheap.
